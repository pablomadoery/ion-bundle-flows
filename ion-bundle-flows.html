    How the architecture works

    Everything is queues + tiny daemons. ION decomposes bundle processing into small single-purpose tasks (ipnfw, bpclm, ltpclo, ltpmeter, ltpcli…) connected by semaphore-signaled queues in the SDR (Spacecraft/Simple Data Recorder), a notionally non-volatile heap. A bundle survives reboots while it waits — essential when contacts are minutes-to-hours apart.

    Zero-copy objects (ZCO). Payload data is written once into the SDR heap; every stage afterwards passes only a reference. A bundle can simultaneously sit in a delivery queue and a transmission queue without duplication; storage is reclaimed when the last reference is released.

    Routing vs. plans. The routing table (nodes, routes, hops — volatile working memory, built from the contact plan via Contact Graph Routing) answers "which neighbor next?". The forwarding database (plans, rules, groups) answers "which outduct/convergence layer reaches that neighbor?".

    LTP aggregation. Multiple bundles are aggregated into one LTP block (amortizing overhead), then ltpmeter cuts the block into link-sized segments. Red-part segments are retransmitted until the receiver's reports show the block is complete — reliability over one long-delay hop, while BP does store-and-forward end-to-end.
